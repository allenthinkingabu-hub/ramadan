# Ramadan AI AI エージェントチーム

**専門化されたAIエージェントのオーケストレーションによる、エンドツーエンドのITプロジェクトデリバリーの自動化**

Ramadan AIは、ソフトウェアデリバリーライフサイクルをコンポーザブルなAIエージェントスキルのシステムに変換します。プロジェクト憲章の策定からリリース後のレトロスペクティブまで、プロジェクトライフサイクルにおけるすべてのタスクは、独自のSOP、準備完了の定義（DoR）、完了の定義（DoD）、RACIマトリクス、およびツール参照を備えた構造化スキルとして体系化されています。マルチエージェントチームがオーケストレーションのもとでこれらのスキルを実行し、各ステージにおいて監査可能かつ品質ゲートを通過した成果物を生成します。

## 動機

エンタープライズITプロジェクトは、インセプション、要件定義、開発、QA、リリースという確立されたライフサイクルフェーズに従います。しかし、各タスクに関する運用知識は、散在するWiki、暗黙知、またはアドホックなチェックリストに依存しているのが現状です。Ramadan AIは以下のアプローチでこの課題を解決します。

1. **デリバリーライフサイクルの分解** — すべてのフェーズにわたり、個別の明確に定義されたタスクに分解します。
2. **各タスクのAIエージェントスキルとしての体系化** — 正式な品質ディメンション（SOP、DoR、DoD、RACI、ツール）を備えたスキルとして定義します。
3. **スキル実行のオーケストレーション** — スーパーバイザー品質ゲートを内蔵したイベント駆動型マルチエージェントアーキテクチャを通じて実行します。

その結果、AIエージェントによって駆動される、再現可能で検査可能、かつ継続的に改善可能なデリバリーシステムが実現されます。

## コアコンセプト

### タスク単位のスキルモデル

ソフトウェアデリバリーライフサイクルにおけるすべてのタスクは、1つのAIエージェントスキルに対応します。スキルは汎用的なプロンプトではなく、以下を含む構造化された実行単位です。

| ディメンション | 目的 | 例 |
|:---|:---|:---|
| **SOP** | 段階的な運用手順（Phase 0〜5） | 技術ディスカバリー評価の実施方法 |
| **DoR** | 実行開始前に満たすべき前提条件 | スコープ定義済み、ステークホルダー特定済み、ツール利用可能 |
| **DoD** | タスク完了前に合格すべき品質チェックリスト | 成果物、プロセス、ドキュメントを網羅する19項目のチェックリスト |
| **RACI** | すべてのステークホルダーに対する責任分担 | SA = 実行責任、PM = 説明責任、TL = 相談先 |
| **ツール** | ツール参照および使用ガイダンス | WebSearch、ファイル操作、ダイアグラム生成ツール |
| **トリガー** | スキルを起動する条件 | 上流タスクの完了、PMからの指示 |
| **出力テンプレート** | 標準化された成果物のフォーマット | レポート構成、意思決定ログのフォーマット |

### スーパーバイザー品質ゲート

すべてのメインスキルには、独立した品質検査を実施する**スーパーバイザースキル**が対になっています。タスクは、スーパーバイザーがすべての検査基準で100%の合格率を達成するまで完了とは見なされません。不合格項目は自動的に修正され、再検査が実施されます。

### イベント駆動型オーケストレーション

エージェント間は構造化されたイベントプロトコルを通じて通信します。

```
PM broadcasts TaskTriggered → Role Agent executes skill → Self-check DoD
  → Supervisor inspects → 100% pass → TaskCompleted reported to PM
  → PM triggers downstream tasks
```

## エージェントチーム

Ramadan AIは、それぞれ異なる役割とスキルポートフォリオを持つ4つのエージェントからなるチームとして運用されます。

| エージェント | 役割 | タイプ | スキル数 | 担当領域 |
|:---|:---|:---|:---:|:---|
| **PM Agent** | プロジェクトマネージャー | オーケストレーター | 24 | 憲章策定、ステークホルダー分析、リソース計画、リスク管理、リリース調整、プロジェクトクローズ |
| **IPM Agent** | ITプロダクトマネージャー | ロールエージェント | 21 | 要件収集、BRD/PRD作成、ユーザーストーリー、受入基準、UAT、ローンチ調整 |
| **SA Agent** | システムアーキテクト | ロールエージェント | 37 | 技術ディスカバリー、アーキテクチャ設計、NFR、インテグレーション設計、セキュリティレビュー、デプロイメントアーキテクチャ |
| **TL Agent** | テクニカルリード | ロールエージェント | 24 | 技術ビジョン、ソリューション設計、コードレビューリーダーシップ、開発標準、技術リスク評価 |

各メインスキルにはスーパーバイザーが対になっており、合計で**212のスキルディレクトリ**（メイン106 + スーパーバイザー106）を構成しています。

## スキル構造

すべてのスキルは標準化されたディレクトリ構成に従います。

```
{role}-{skill-name}/
├── SKILL.md                        # Phase 0-5ワークフローを含むスキル定義
└── references/
    ├── sop.md                      # 標準業務手順書
    ├── dor.md                      # 準備完了の定義
    ├── dod.md                      # 完了の定義
    ├── raci.md                     # RACIマトリクス
    ├── tools.md                    # ツール使用リファレンス
    ├── triggers.md                 # 起動条件
    ├── output-templates.md         # 成果物テンプレート
    └── skills-and-knowledge.md     # 必要なコンピテンシー
```

スーパーバイザースキルには、`inspection-criteria.md` ファイルおよび `scripts/verify_dod.py` 自動検証スクリプトが追加されます。

### 汎用Phase 0〜5ワークフロー

すべてのスキルは一貫性を確保するため、同一の6フェーズワークフローを実行します。

| フェーズ | 名称 | 目的 |
|:---:|:---|:---|
| 0 | 初期化 | 出力ディレクトリの作成、ログの初期化、DoRの検証 |
| 1 | タスク目的の理解 | 対話を通じた目的の明確化、ユーザー確認の取得 |
| 2 | トピックの理解 | ドメインコンテキストの深掘り、背景情報の収集 |
| 3 | リサーチ＆Q&A | 業界リサーチ、反復的な質問生成、専門家への相談 |
| 4 | 実行＆デリバリー | テンプレートを使用した成果物の作成、DoDに基づくセルフチェック |
| 5 | 完了＆ハンドオフ | スーパーバイザーの起動、不合格項目の修正、PMへの報告 |

## デリバリープレイブック

[`delivery_playbook.md`](task/delivery_playbook.md) は、5つのプロジェクトフェーズにわたるタスク実行順序の全体像を、順次実行されるウェーブとして定義しています。

### フェーズとタスク分布

| フェーズ | 概要 | 主要活動 |
|:---|:---|:---|
| **インセプション** | プロジェクトの立ち上げと実現可能性評価 | 憲章策定、ステークホルダー分析、要件収集、技術選定、リスク特定 |
| **要件定義** | 詳細な仕様策定と設計 | アーキテクチャ設計、NFR定義、インテグレーション設計、データアーキテクチャ、PRD作成、技術標準 |
| **開発** | 実装と技術ガイダンス | コード/設計レビュー、スパイク/PoCリーダーシップ、技術的負債管理、ADRドキュメント作成 |
| **QA** | 検証とコンプライアンス | パフォーマンステスト、セキュリティレビュー、コンプライアンス検証、インフラストラクチャ検証、UAT調整 |
| **リリース** | デプロイメントと安定化 | デプロイメントアーキテクチャ、監視設定、キャパシティプランニング、ゴーライブ調整、ハイパーケア、レトロスペクティブ |

### ウェーブ実行モデル

同一ウェーブ内のタスクは並行して実行されます。ウェーブNのすべてのタスクが完了するまで、ウェーブN+1は開始されません。これにより、タスクの依存関係を尊重しつつ、最大限の並列性が実現されます。

## 実行モデル

### イベントプロトコル

| イベント | 方向 | ペイロード |
|:---|:---|:---|
| `TaskTriggered` | PM → ロールエージェント | `{ task_id, skill_dir, inputs, context }` |
| `TaskCompleted` | ロールエージェント → PM | `{ task_id, status, artifacts, supervisor_report }` |
| `SupervisorTriggered` | ロールエージェント → スーパーバイザー | `{ task_id, output_dir }` |
| `SupervisorCompleted` | スーパーバイザー → ロールエージェント | `{ task_id, pass_rate, report, remediation_items }` |

### 品質保証フロー

```
Role Agent completes skill execution
       │
       ▼
Self-check: Verify all 19 DoD items
       │
       ├── FAIL → Auto-remediate, re-verify
       │
       ▼
Invoke paired Supervisor skill
       │
       ├── FAIL → Receive remediation report, fix, re-invoke supervisor
       │
       ▼
100% pass rate achieved → Report TaskCompleted to PM
```

### 冪等性

すべてのタスク実行は冪等です。システムはタスクの状態（`PENDING`、`IN_PROGRESS`、`DONE`、`FAILED`）を追跡し、適切に再開またはスキップすることで、重複作業を防止し、信頼性の高いリカバリーを可能にします。

- Python 3.8以上
- Claude Code CLI（Claude Opus 4.6モデルへのアクセス権が必要）
- Git

1. **リポジトリのクローン**
   ```bash
   git clone <repository-url>
   cd ramadan
   ```

2. **チームパッケージのインストール**
   ```bash
   python scripts/install_team.py
   ```

3. **エージェントワークスペースのブートストラップ**
   ```bash
   python scripts/bootstrap_workspaces.py
   ```

4. **スキル構造のバリデーション**
   ```bash
   python openclaw-team/scripts/quick_validate.py
   ```

スキルスキャフォールディングツールを使用して、必要なすべてのリファレンスファイルを含む新規スキルを生成します。

このコマンドにより、SOP、DoR、DoD、RACI、ツール、トリガー、出力テンプレートのテンプレートファイルを含む完全なディレクトリ構成が作成されます。

## 詳細解説: プロジェクト構造スキャンとエージェントメモリ

**プロジェクト構造スキャン**スキル（`project-structure-scan`）は、Ramadan AIにおいて最もアーキテクチャ的に高度なスキルです。このスキルは、ほとんどのAIエージェントスキルに欠けている重要な能力のリファレンス実装として機能します。それは、**セッションをまたいで永続化し、プロジェクト間で知識を伝達する構造化されたメモリ**です。

### 課題: ステートレスなAIエージェント

デフォルトでは、AIエージェントはステートレスです。すべての会話はゼロから始まります。エージェントが今日コードベースをスキャンし、ユーザーが翌週に同じプロジェクトのスキャンを依頼しても、エージェントには以前の発見に関する記憶がなく、何が変更されたかの認識もなく、冗長な作業をスキップする能力もありません。これにより以下の問題が発生します。

- 変更のないコードの繰り返し分析
- ユーザーが既に回答した質問の再質問
- プロジェクトの時間的変化を追跡する能力の欠如
- プロジェクト間の学習不在（プロジェクトAの知見がプロジェクトBに活用されない）

プロジェクト構造スキャンスキルは、エージェントの呼び出しごとに段階的にインテリジェンスを向上させる**SQLiteベースのメモリシステム**でこの課題を解決します。

### メモリアーキテクチャ

メモリシステムは、ローカルのSQLiteデータベース（`memory/agent_memory.db`）に4つの専用テーブルを使用します。

```
┌─────────────────────────────────────────────────────────────┐
│                    agent_memory.db                           │
│                                                             │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────────┐  │
│  │ task_memory  │  │ knowledge    │  │   scan_history    │  │
│  │             │  │ _base        │  │                   │  │
│  │ findings    │  │              │  │ session tracking  │  │
│  │ decisions   │  │ patterns     │  │ scope & status    │  │
│  │ lessons     │  │ dependencies │  │ deliverables path │  │
│  │ questions   │  │ tech_stack   │  │ timestamps        │  │
│  │ risks       │  │ conventions  │  │                   │  │
│  │             │  │ insights     │  │                   │  │
│  │ (per phase, │  │              │  │                   │  │
│  │  per session)│  │ (confidence  │  │                   │  │
│  │             │  │  scored,     │  │                   │  │
│  │             │  │  cross-      │  │                   │  │
│  │             │  │  project)    │  │                   │  │
│  └─────────────┘  └──────────────┘  └───────────────────┘  │
│                                                             │
│  ┌─────────────┐                                            │
│  │ dod_checks  │    Supplementary Files:                    │
│  │             │    ├── logs/conversation-log.md             │
│  │ per-round   │    ├── logs/work-log.md                    │
│  │ pass/fail   │    ├── research/{topic}.md                 │
│  │ evidence    │    └── phase{N}-questions.md                │
│  └─────────────┘                                            │
└─────────────────────────────────────────────────────────────┘
```

| テーブル | 目的 | 主要フィールド |
|:---|:---|:---|
| `task_memory` | セッション単位の発見事項、意思決定、教訓、Q&A、リスク | `session_id`, `phase`, `memory_type`, `content`, `tags` |
| `knowledge_base` | 信頼度スコア付きのセッション横断型再利用可能ナレッジ | `category`, `key`, `value`, `confidence` (0.0〜1.0) |
| `scan_history` | すべてのスキャン実行の監査証跡 | `project_name`, `scan_scope`, `status`, `deliverables_path` |
| `dod_checks` | ラウンドごとの品質ゲート検証結果 | `check_round`, `check_item`, `status`, `evidence` |

### メモリとワークフロー各フェーズの統合

メモリシステムは後付けの機能ではなく、スキル実行のすべてのフェーズに組み込まれています。

**Phase 0 — メモリ駆動型初期化**

起動時に、エージェントは `scan_history` を照会し、このプロジェクトが以前スキャンされたかどうかを判定します。この単一のクエリが実行パス全体を決定します。

```
load_project_history(project_name)
  │
  ├── Previous scans found → Re-scan Mode
  │   └── generate_memory_summary() → present to user
  │       └── User chooses: Incremental / Full re-scan / Review previous
  │
  ├── No history, but similar projects exist → Hypothesis Mode
  │   └── get_similar_projects(tech_stack_tags)
  │       └── "Project X used DDD with Hexagonal. Look for similar patterns?"
  │
  └── No history, no similar projects → Fresh Scan Mode
```

作業開始前に、エージェントはメンテナンス操作も実行します。`apply_confidence_decay()` は90日間更新されていないナレッジエントリの信頼度スコアを減衰させ、`prune_low_confidence()` は0.3の閾値を下回ったエントリを削除します。

**Phase 1 — メモリ強化型目的理解**

毎回ゼロから「なぜこのスキャンが必要ですか？」と質問する代わりに、エージェントは以前の目的記録を照会します。

```python
load_lessons_learned(task_id='SA-DISC-001', project_name=...) # filtered by phase='phase1'
```

記録が存在する場合、「前回このプロジェクトをスキャンした際の目的は{previous_purpose}でした。今回も同じ目的ですか？」と提示し、5分間の対話を1回の確認に短縮します。

**Phase 2 — メモリ加速型ディスカバリー**

エージェントは `knowledge_base` から以前記録された技術スタック、パターン、規約を照会します。

```python
load_project_knowledge(project_name)  # returns entries sorted by confidence DESC
```

既知の回答は事前に入力されます。未知の項目や変更の可能性がある項目、すなわちデルタ質問のみが質問されます。再スキャン時には、質問の60〜80%が削減されます。

**Phase 3 — メモリ活用型リサーチ**

以前のリサーチファイルとリスク記録がロードされます。エージェントは冗長なリサーチをスキップし、ギャップに焦点を当て、既知のリスクを積極的に再確認します。「前回、以下のリスクを特定しました: {list}。これらが解決済みかどうか確認しますか？」

**Phase 4 — メモリ最適化型実行**

増分スキャンでは、エージェントは以前の `OUT-01`（プロジェクト構造ツリー）をロードし、現在のディレクトリ構造との差分を取得して、変更された領域のみを詳細スキャンします。以前の `OUT-04`（パッケージ依存関係マップ）は現在のマニフェストと比較され、変更のみの依存関係レポートが生成されます。これにより、再スキャン時の実行時間が40〜70%短縮されます。

### プロジェクト間ナレッジトランスファー

最も強力なメモリ機能はプロジェクトの境界を超えて動作します。エージェントが新しいプロジェクトに遭遇した際、類似の技術スタックを持つプロジェクトをナレッジベースから検索します。

```python
get_similar_projects(db_path, tech_stack_tags=["spring-boot", "postgresql", "redis"])
# Returns: [{"project_name": "ProjectX", "matching_tags": [...], "match_count": 3}]
```

類似プロジェクトからの高信頼度パターンは、新規プロジェクトに対する仮説となります。エージェントが過去に3つのSpring Bootプロジェクトすべてでヘキサゴナルアーキテクチャパターンが使用されていることを特定していた場合、「記録内の類似プロジェクトではヘキサゴナルアーキテクチャが使用されています。このプロジェクトでも同様のパターンを探しますか？」と積極的に提案でき、初回スキャン分析を約30%加速します。

### 信頼度減衰と矛盾検出

ナレッジは永久に有効であるわけではありません。メモリシステムは整合性を維持するために2つのメカニズムを実装しています。

1. **信頼度減衰**: 90日ごとに、更新されていないエントリの信頼度ポイントが0.2減少します（最小値0.0）。信頼度0.8で記録されたエントリは、再確認されない場合、0.6 → 0.4 → 0.2 → 0.0と連続する期間にわたって減衰します。0.3を下回ったエントリは削除され、ユーザーに報告されます。

2. **矛盾検出**: 新しいナレッジエントリを書き込む前に、エージェントは既存エントリとの競合をチェックします。

   ```python
   contradiction = detect_contradictions(db_path, project_name, category, key, new_value)
   # Returns existing vs. new value if they differ
   ```

   矛盾が検出された場合（例: アーキテクチャパターンが以前「MVC」と記録されていたが、新しいエビデンスでは「ヘキサゴナル」を示唆している場合）、エージェントは競合をフラグとして提示し、暗黙的に上書きするのではなく、ユーザーに解決を求めます。

### 設計原則

メモリアーキテクチャは、以下の意図的な設計方針に基づいています。

1. **クラウドストレージではなくSQLiteを採用**: メモリはローカルでポータブルであり、インフラストラクチャを一切必要としません。データベースファイルはリポジトリにコミットしたり、チームメンバー間で共有したりすることができます。

2. **二値的な真偽ではなく信頼度スコアリング**: ナレッジは単純に「既知」か「未知」かではなく、時間の経過とともに減衰する信頼度スコアを持ちます。これは、ソフトウェアプロジェクトが進化し、昨日の真実が今日も有効とは限らないという現実を反映しています。

3. **UPSERTセマンティクス**: `record_knowledge()` はUPSERTを実行します。同一の `(project_name, category, key)` タプルが既に存在する場合、重複エントリを作成するのではなく、値と信頼度が更新されます。

4. **タスクメモリとナレッジの分離**: `task_memory` はセッションスコープであり、単一実行の生のQ&A、意思決定、発見事項を記録します。`knowledge_base` はプロジェクトスコープであり、セッションをまたいで永続化される蒸留済みの再利用可能なナレッジを記録します。

5. **WALジャーナルモード**: データベースはWrite-Ahead Loggingを使用しており、書き込み中の読み取りをブロックすることなく、優れた同時実行性を実現します。

## コントリビューション

### スキルの追加

1. `task/` にエージェントスキル定義ドキュメントとしてタスクを定義します。
2. プレイブック内のデリバリーフェーズとウェーブにタスクをマッピングします。
3. `init_skill.py` を使用してスキルをスキャフォールディングします。
4. すべてのリファレンスファイル（SOP、DoR、DoD、RACI、ツール、トリガー、出力テンプレート）を作成します。
5. 検査基準を含む対となるスーパーバイザースキルを作成します。
6. `config/openclaw.json` にスキルを登録します。
7. `quick_validate.py` でバリデーションを実行します。

### スキル品質基準

すべてのスキルには以下が含まれている必要があります。
- フェーズの省略がないPhase 0〜5の完全なSOP
- 検証可能な前提条件を含むDoRチェックリスト
- 19項目の品質ゲートを含むDoDチェックリスト
- 関連するすべてのステークホルダーを割り当てたRACIマトリクス
- すべての成果物に対する出力テンプレート
- 検査基準を含む対となるスーパーバイザースキル
